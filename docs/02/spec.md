# MewCode 第一阶段 Spec

## 背景

MewCode 是一个从零开始构建的命令行 AI 编程助手，目标形态对标 Claude
Code——在终端里通过自然语言对话完成开发任务。完整的 MewCode 需要包含工
具调用、文件读写、代码编辑、上下文管理等 agent 能力，工程量较大，因此
拆分为多个迭代阶段。

本阶段（第一阶段）只交付**纯对话能力**，作为后续 agent 功能的运行底座。
完成本阶段后，用户在终端启动 MewCode 即可进入交互式 REPL，与大模型多
轮对话；流式逐字打印；支持 Anthropic 协议（含 extended thinking）和
OpenAI 协议两种后端，通过项目级 YAML 配置文件管理多个命名供应商。

工具调用、文件操作、代码编辑、会话持久化等能力**不在本阶段范围内**，
留待后续迭代。

## 目标

- 提供一个可在终端运行的交互式 REPL，用户输入即获 AI 流式回复
- 在单进程内支持多轮对话，AI 能引用同一会话内的历史上下文
- 通过统一的 Provider 抽象层，同时支持 Anthropic 协议与 OpenAI 协议
  两种后端，配置文件切换无需改代码
- 在 Anthropic 协议下支持 Claude 风格的 extended thinking，思考过程
  以视觉区分的方式流式展示
- 通过项目级 YAML 配置文件管理多个命名供应商，运行时可在已配置的供应
  商之间切换
- 提供必要的内置控制命令（退出、清空、切换供应商、开关思考等）
- 为后续阶段（工具调用、文件操作）预留清晰的扩展点

## 功能需求

### 启动与配置

- **F1：项目级 YAML 配置加载**
  启动时从当前工作目录读取 `mewcode.yaml`。配置文件不存在、解析失败、
  或缺失必需字段时，向用户输出明确的错误信息并退出，不进入 REPL。

- **F2：命名供应商列表**
  配置文件支持声明多个命名供应商条目，每个条目包含四个核心字段：
  protocol（取值 `anthropic` 或 `openai`）、model、base_url、api_key。
  配置文件顶层通过 `default` 字段指定启动时使用的供应商名。
  `default` 指向的名称必须存在于供应商列表中，否则启动失败并报错。

- **F3：启动信息提示**
  REPL 启动后向用户展示当前生效的供应商名、协议、模型，以及可用斜杠命
  令的简短提示。

### 交互式 REPL

- **F4：多行输入与提交**
  提供基于交互式输入区，支持光标移动、行内编辑、历史回溯
  （上/下方向键调出本次会话内输入过的历史 prompt）。回车提交一条用户
  消息。

- **F5：流式渲染回复**
  AI 回复以流式方式逐字（逐 chunk）打印到终端。打印过程中支持 Markdown
  常见元素的可读呈现（标题、列表、行内代码、代码块、加粗）。同一条回
  复打印完毕后追加换行，回到输入提示符。

- **F6：进程内多轮对话**
  在单次 MewCode 进程生命周期内维护一份消息列表，按用户消息与 AI 回复
  交替追加，每次请求时携带完整历史发送给后端，使 AI 能引用之前轮次的
  内容。进程退出后历史不持久化。

### Provider 抽象与协议支持

- **F7：协议级 Provider 抽象**
  系统提供一个统一的 Provider 抽象，定义"发送一段对话历史并以流式方式
  返回回复事件"的能力。具体实现按 wire protocol 划分：一个负责
  Anthropic 协议（/v1/messages SSE），一个负责 OpenAI 协议
  （/v1/chat/completions SSE）。配置中的 `protocol` 字段决定加载哪个
  实现。

- **F8：扩展点**
  新增协议（如未来加入 Gemini 原生协议）只需新增一个 Provider 实现并
  在协议分发处注册，不需要修改 REPL、配置加载、对话状态管理等其他模块。

### Extended Thinking

- **F9：思考开关**
  系统维护一个布尔状态"思考开启/关闭"，默认为关闭。仅当当前供应商
  protocol 为 `anthropic` 时该状态才生效；为 `openai` 时该状态被忽略，
  且在用户尝试开启时给出提示说明当前协议不支持。

- **F10：思考流式展示**
  思考开启时，Anthropic Provider 在请求中携带 thinking 参数；返回流中
  的思考内容以视觉上明显区分的样式（灰色、斜体）流式打印，并在思考开
  始与结束位置带有可识别的标记（如 `▎思考中…` 起始标记、思考结束后空
  行分隔），随后以正常样式流式打印最终回复。

### 内置斜杠命令

- **F11：命令识别与分发**
  以 `/` 开头的输入被识别为命令，不发送给大模型；命令未注册时给出"未
  知命令"提示并列出可用命令。

- **F12：必备命令集**
  实现以下命令：
  - `/exit` 与 `/quit`：退出 MewCode
  - `/help`：列出所有可用命令及简短说明
  - `/clear`：清空当前会话的消息历史（发送给模型的对话上下文），开始
    新对话；不退出进程。输入历史（方向键回溯的已提交 prompt 列表）不
    受影响
  - `/think on` 与 `/think off`：开启/关闭 extended thinking
  - `/providers`：列出配置文件中所有已声明的供应商名（标记当前生效项）
  - `/provider <name>`：切换到指定供应商；切换时清空当前会话历史，并
    向用户展示切换结果（新供应商名、协议、模型）

### 用量与错误反馈

- **F13：Token 用量展示**
  每次 AI 回复完整结束后，在回复下方追加一行灰色文字展示本次调用的
  token 用量：输入 token、输出 token、思考 token（仅当 thinking 开启
  且当前协议为 anthropic 时显示思考项）。具体数值取自 Provider 返回
  的 usage 数据；后端未返回 usage 时该行省略。

- **F14：错误直报**
  Provider 调用过程中产生的网络错误、HTTP 非 2xx、API 鉴权失败、SSE 流
  解析异常等，均以红色文字打印明确的错误信息（包含错误类别与原始消息），
  随后回到输入提示符等待下一条输入。本阶段不做任何形式的自动重试。

## 非功能需求

### 性能与响应

- **N1：流式首字时延**
  用户提交消息后，从 Provider 收到首个内容字节起即刻开始向终端渲染，
  不缓冲到段落或整段回复结束才输出。

- **N2：流式不阻塞输入**
  AI 流式输出过程中，渲染逻辑不得长时间占用主循环导致终端无响应；
  Ctrl+C 应能在合理时间内（秒级）中止当前流式输出并回到输入提示符。

### 可用性

- **N3：明确的错误提示**
  所有用户可见的错误信息（配置错误、网络错误、协议错误、未知命令等）
  必须包含错误类别和具体原因，避免出现仅"出错了"之类无信息量的提示。

- **N4：终端兼容性**
  REPL 应在主流终端（Windows Terminal、cmd.exe、PowerShell、
  Linux/macOS 常见终端模拟器、tmux 会话）下正常工作。颜色与样式在不
  支持 ANSI 的终端下应自动降级为纯文本，不出现乱码或控制字符泄漏。

- **N5：Ctrl+C 与退出语义**
  在输入空白时按 Ctrl+C 给出"再按一次或输入 /exit 退出"提示；连续两
  次 Ctrl+C 退出进程。在 AI 流式输出过程中按 Ctrl+C 中止当前流式输出，
  已打印到终端的部分保留显示但整条 AI 回复不进入会话历史（视为本轮请
  求作废），随后回到输入提示符等待下一条输入。

### 可维护性

- **N6：模块边界清晰**
  配置加载、Provider 抽象与实现、对话状态管理、命令分发、TUI 渲染
  各自独立，模块间通过明确定义的接口或数据结构交互，禁止跨模块直接
  访问内部状态。

- **N7：协议扩展成本**
  新增一种 wire protocol 的 Provider 实现，所需改动应限定在新增一个
  Provider 文件 + 在协议分发处注册一行，不应触及配置加载、REPL、命
  令分发等其他模块。

- **N8：中文注释**
  所有源码中的注释、文档字符串、用户可见的提示文本均使用中文。

### 安全与合规

- **N9：敏感信息保护**
  api_key 不得出现在任何用户可见的输出中，包括启动信息、错误信息、
  日志、token 用量行等。配置加载失败的报错可指出"api_key 字段缺失/
  为空"，但不得回显其值。

### 兼容性

- **N10：Python 版本**
  支持 Python 3.10 及以上版本运行。

- **N11：依赖最小化**
  本阶段所引入的第三方依赖应限定在"TUI 输入/渲染、YAML 解析、流式
  HTTP 客户端"这三类基础设施所必需的最小集合内；不引入 Anthropic 或
  OpenAI 厂商官方 SDK——Provider 实现直接基于各自 wire protocol 的
  SSE 进行，以保证对 base_url 的完全控制和对兼容后端的支持。具体库
  选型在 plan.md 中确定。

## 不做的事

本阶段明确**不**包含以下能力，留待后续迭代：

### Agent 能力（核心后续工作）

- 不做工具调用（tool use / function calling）；请求体不包含 tools 字段
- 不做文件读写；AI 不能读取、写入、编辑用户文件系统
- 不做代码执行；AI 不能执行 shell 命令、运行代码片段、调用外部进程
- 不做项目上下文注入；不扫描当前工作目录、不自动读取 README/源码塞进
  prompt

### 会话与持久化

- 不做对话历史持久化；进程退出后历史消失
- 不做跨会话记忆；没有"长期记忆"或向量召回

### Provider 与配置

- 不引入厂商官方 SDK（anthropic-sdk-python、openai 等）
- 不读取用户级配置文件（如 `~/.mewcode/config.yaml`）；只认当前工作目
  录的 `mewcode.yaml`
- 不做环境变量回填；即便检测到 `ANTHROPIC_BASE_URL` 等环境变量也不会
  作为 fallback
- 不做配置热加载；运行时修改 `mewcode.yaml` 不生效，需重启

### 错误处理与可靠性

- 不做自动重试
- 不做断线重连；SSE 流中断即视为本次请求失败

### 高级 TUI 能力

- 不做多面板/全屏 TUI
- 不做用户输入区的语言级语法高亮
- 不做图片/附件输入

### 用量与计费

- 不做成本估算；只展示 token 数字，不维护单价表

### 系统提示

- 不设默认 system prompt；请求体不携带 system 字段；配置文件不支持
  `system_prompt` 字段

### 模型能力

- 不做运行时模型参数调节命令（temperature、top_p、max_tokens 等）
- 不做 thinking budget 调节命令

### 多语言/国际化

- 不做 i18n；界面文案与错误信息全部为中文

## 验收标准

每条 AC 通过运行代码或观察终端行为来验证；与具体实现解耦。

### 启动与配置

- **AC1：合法配置启动成功**
  在工作目录放置一份合法的 `mewcode.yaml`（含至少一个供应商 + 合法
  default），启动 MewCode 后能进入 REPL 输入提示符，并打印当前生效的
  供应商名、协议、模型。

- **AC2：缺失配置文件报错**
  在不含 `mewcode.yaml` 的目录启动 MewCode，进程不进入 REPL，终端打
  印明确错误信息指出"配置文件不存在"，进程以非 0 退出码结束。

- **AC3：非法配置报错**
  使用一份缺失必需字段（如缺 `protocol` 或 `default` 指向不存在的供
  应商）的 `mewcode.yaml` 启动，终端打印明确错误信息指出问题字段，进
  程以非 0 退出码结束。

- **AC4：api_key 不回显**
  无论配置正确还是错误，终端任何输出（启动信息、错误信息、token 用量
  行）中都不出现 api_key 的实际值。

### 多轮对话与流式

- **AC5：流式逐字打印**
  向 MewCode 发送一条会让 AI 产生较长回复的提问（例如"用一段话介绍
  Python"），AI 回复在终端逐字（逐 chunk）出现，而不是等整段生成完毕
  后一次性打印。

- **AC6：Markdown 渲染**
  让 AI 输出含标题、列表、代码块的 Markdown 内容，终端展示具有可读的
  样式区分（标题加粗、代码块带框/底色），未出现原始 `#`、` ``` ` 等
  Markdown 字符堆积。

- **AC7：多轮上下文记忆**
  连续进行三轮对话：第 1 轮告诉 AI 一个事实（如"我叫小明"），第 2 轮
  问无关问题，第 3 轮让 AI 复述第 1 轮的事实，AI 能正确复述。

- **AC8：输入历史回溯**
  在 REPL 输入区按上方向键，能依次调出本次会话内已提交过的历史 prompt。

### Provider 抽象与协议切换

- **AC9：Anthropic 协议可用**
  在配置中将 default 指向一个 protocol 为 `anthropic` 的供应商（如题
  目给出的 DeepSeek-via-Anthropic），启动后正常完成多轮对话，AI 回复
  通过 SSE 流式到达。

- **AC10：OpenAI 协议可用**
  在配置中将 default 指向一个 protocol 为 `openai` 的供应商，启动后
  正常完成多轮对话，AI 回复通过 SSE 流式到达。

- **AC11：运行时切换供应商**
  在已含至少两个供应商的配置下启动，使用 `/providers` 列出供应商，使
  用 `/provider <name>` 切换到另一个，终端打印切换结果，会话历史被清
  空，后续对话使用新供应商。

### Extended Thinking

- **AC12：默认关闭，回复中无思考块**
  默认启动状态下（thinking 关闭）发起对话，终端只显示 AI 最终回复，
  无灰色斜体的思考内容，无 `▎思考中…` 标记。

- **AC13：开启后思考流式展示**
  当前供应商为 anthropic 协议时，执行 `/think on`，发起一条复杂提问，
  终端先以灰色斜体流式打印思考内容（带起始标记 `▎思考中…`），思考结
  束后空行分隔，再以正常样式流式打印最终回复。

- **AC14：OpenAI 协议下开启思考给出提示**
  当前供应商为 openai 协议时，执行 `/think on`，终端打印提示说明当前
  协议不支持思考；后续对话不携带 thinking 参数。

### 内置斜杠命令

- **AC15：/help 列出全部命令**
  执行 `/help`，终端打印 `/exit /quit /help /clear /think /provider
  /providers` 全部命令及其简短说明。

- **AC16：/clear 清空历史**
  对话若干轮后执行 `/clear`，再发起一条只能依赖历史才能回答的问题（如
  "我刚才说我叫什么？"），AI 回复表明不知道。

- **AC17：未知命令提示**
  输入 `/foobar`，终端打印"未知命令"提示并列出可用命令列表，不向大模
  型发送请求。

- **AC18：/exit 与 /quit 退出**
  分别测试 `/exit` 和 `/quit`，进程均能正常退出，退出码为 0。

### 用量展示

- **AC19：回复后展示 token 用量**
  完成一次对话后，在 AI 回复下方出现一行灰色文字，包含输入 token 数和
  输出 token 数。开启 thinking 且协议为 anthropic 时，该行额外包含思
  考 token 数。

### 错误处理

- **AC20：错误以红字明确报告**
  人为构造一个错误场景（如把 api_key 改为非法值），发起一条对话，终
  端以红色文字打印明确的错误信息（包含错误类别与具体原因），不发生重
  试，随后回到输入提示符可继续输入或修改配置。

### 中断语义

- **AC21：流式中断不进历史**
  发起一条会让 AI 产生长回复的提问，在 AI 流式输出过程中按 Ctrl+C，
  当前流式输出立即停止，已打印的内容保留在终端可见，回到输入提示符。
  下一轮发送"刚才你说了什么？"，AI 回复表明不记得（因为被中断的回复
  未进入历史）。

- **AC22：空输入下连按两次 Ctrl+C 退出**
  在空白输入提示符下按 Ctrl+C，终端提示"再按一次或输入 /exit 退出"，
  再按一次 Ctrl+C，进程退出。

### 终端兼容性

- **AC23：tmux 端到端运行**
  在 tmux 会话中启动 MewCode，完成上述任意一组多轮对话场景，颜色、
  Markdown 样式、流式渲染、token 用量行均正常显示，无控制字符泄漏。

### 扩展性（结构性验收）

- **AC24：协议扩展性可见**
  通过阅读代码确认：要新增一种 wire protocol 的 Provider 实现，所需
  改动限定在新增一个 Provider 文件 + 在协议分发处注册一行；REPL、配
  置加载、命令分发、对话状态管理模块的代码均无需修改。
