# MewCode 第二阶段 Spec

## 背景

MewCode 第一阶段已交付纯对话 REPL（docs/02/），用户可在终端与 LLM
进行多轮流式对话，支持 Anthropic 与 OpenAI 两种协议、命名供应商切换、
extended thinking 等。但模型只能"动嘴"——不能读文件、不能写代码、
不能跑命令，无法成为真正的编程助手。

本阶段（第二阶段）为 MewCode 装上**工具系统**：模型可以请求调用工具
（read / write / edit / run / glob / search 共 6 个核心工具），由
MewCode 在本地执行后把结果回灌进对话历史，模型据此组织最终回复。
从此 MewCode 从"聊天机器人"演化为"能干活的 Agent 雏形"。

本阶段实现**"一次工具 + 一次答复"的单轮闭环**：用户每次输入触发
最多两次 LLM 请求——

```
用户 prompt
        ↓
    LLM 请求 #1（流式）→ 可能的 text + tool_use × N
        ↓
    串行执行所有 tool_use（写类/命令类执行前向用户征求确认）
        ↓
    tool_results × N 追加到历史
        ↓
    LLM 请求 #2（流式）→ 最终文本答复
        ↓
    回到用户输入提示符（等待下一条 prompt）
```

如果 LLM 请求 #1 直接给出文本答复（不含 tool_use），则不发起请求 #2，
行为退化为第一阶段的纯对话。如果 LLM 请求 #2 的回复中再次包含
tool_use，**本阶段硬停**：text 部分写入历史正常显示，tool_use 块
丢弃并在终端打一行灰字提示"模型在最终答复中还想调用工具，本阶段不再
继续；下一轮可以追问"。多工具的真正循环（Agent Loop）留待后续阶段。

## 目标

- 提供统一的 Tool 抽象层（name、description、参数 Schema、execute），
  6 个核心工具均实现该抽象，便于后续扩展
- 提供工具注册中心，按名查找、按目标协议格式输出工具元信息
- 在 Anthropic 与 OpenAI 两种协议下都能完整支持工具调用：请求体携带
  tools 字段、流式 SSE 中正确解析工具调用事件（含 JSON 参数碎片拼接）
- 实现"一次工具 + 一次答复"的单轮闭环：执行所有 tool_use → 把
  tool_results 拼回历史 → 发起 Round 2 请求 → 流式渲染最终答复
- Round 2 若再含 tool_use，硬停不执行；text 部分仍正常入历史与显示，
  额外一行灰字提示
- 工具执行带超时与结构化错误处理，失败信息以 tool_result 包回模型，
  让模型有机会调整而不是让进程崩溃
- 6 个工具的语义满足实际开发需求：读文件、写文件、按原文唯一匹配
  替换改文件、跨平台执行命令、按 glob 模式查找文件、用正则搜索代码
- 路径相关工具受工作目录边界保护，越界返回结构化错误
- 危险操作（写文件、改文件、执行命令）执行前向用户征求确认；用户
  拒绝时把"用户拒绝执行"作为 tool_result 反馈给模型，进入 Round 2
- 工具执行过程在 REPL 中可见：调用前展示工具名与关键参数，结果出炉后
  展示简略反馈（如行数、匹配数、退出码）
- 为后续阶段（Agent Loop、子代理、更多工具）预留清晰的扩展点

## 功能需求

### Tool 抽象与注册

- **F1：Tool 抽象基类**
  系统提供一个统一的 Tool 抽象，每个具体工具都实现该抽象。Tool 必须
  对外暴露：唯一名称、面向模型的简要描述、参数的 JSON Schema 定义、
  以及一个异步执行入口，接收已校验的参数并返回结构化执行结果。

- **F2：工具注册中心**
  系统维护一个工具注册中心，集中登记所有可用工具。注册中心提供：按名
  称查找工具、列出全部已注册工具、把已注册工具按目标协议格式（Anthropic
  或 OpenAI）批量序列化为协议要求的工具元信息列表。新增工具时只需在
  注册中心追加一行注册即可，不需要修改 Provider、REPL、对话引擎等
  其他模块。

- **F3：工具执行结果结构**
  工具执行返回结构化结果：执行是否成功、面向模型的文本输出、可选的
  错误类别。任何执行期间发生的异常（参数非法、文件不存在、命令失败、
  超时等）都被工具自身捕获并转换为"成功=false + 错误描述"的结构化
  结果，而不是抛异常导致进程崩溃。

### 6 个核心工具

- **F4：读文件工具（read）**
  接收文件路径；可选参数指定按行的起始行号（1-based）与读取行数。
  默认行为读取整个文件内容；当文件超过 256KB 时截断到 256KB 并在结果
  中明确标注"已截断"。文件不存在、路径越界、非文本编码无法解码等
  情况返回结构化错误。

- **F5：写文件工具（write）**
  接收文件路径与完整内容；目标文件不存在则新建（含必要的父目录），
  存在则整体覆盖。执行前向用户征求确认（展示路径与内容前若干行）；
  用户拒绝时返回"用户拒绝执行"结构化结果。路径越界返回结构化错误。

- **F6：改文件工具（edit）**
  接收文件路径、原文片段、新文片段。执行时读取目标文件内容,要求
  "原文片段"在文件中按字节精确匹配出现且仅出现一次,命中即用"新文
  片段"替换；不匹配或匹配多次时返回明确的结构化错误（"未找到匹配"
  或"匹配 N 次,请提供更多上下文使匹配唯一"），让模型据此重试。
  执行前向用户征求确认（展示路径与基于 diff 的摘要）。

- **F7：执行命令工具（run）**
  接收一条命令字符串，在启动时的 CWD 下用系统 shell 执行。返回标准
  输出、标准错误、退出码。执行前向用户征求确认（展示完整命令）。
  超出超时（60 秒）则强制终止子进程并返回"超时"结构化错误。stdout +
  stderr 总输出超过 32KB 时截断并标注。

- **F8：glob 工具**
  接收 glob 模式字符串（如 `**/*.py`），在 CWD 下递归匹配文件路径。
  结果中自动排除常见噪声目录（`.git`、`__pycache__`、`node_modules`、
  `.venv`、`dist`、`build`、`.pytest_cache`、`.mypy_cache`、
  `*.egg-info`）。返回的文件路径列表按字母序排序。具体实现技术由
  plan.md 决定。

- **F9：搜索代码工具（search）**
  接收正则表达式（默认）或字面量字符串、可选的文件 glob 过滤、可选的
  匹配上下文行数。在 CWD 下递归搜索文件内容，返回每条匹配的：文件路
  径、行号（1-based）、行内容（单条匹配行截断到 500 字符）。同样
  自动排除噪声目录。

### 工作目录边界

- **F10：路径边界校验**
  路径相关工具（read / write / edit / glob / search）所有路径必须落在
  启动时的 CWD 及其子目录内；越界（含通过 `..` 上溯到父目录、绝对
  路径指向 CWD 外）时拒绝执行并返回结构化错误"路径越界：<path> 不在
  工作目录 <cwd> 内"。run 工具以 CWD 为子进程工作目录运行。

### 单轮闭环：从工具调用到最终答复

- **F11：协议层工具调用支持**
  Anthropic 与 OpenAI 两种协议的 Provider 都需支持工具调用：请求体
  按各自协议格式携带 tools 列表；流式 SSE 中正确解析工具调用事件，
  把 JSON 参数碎片按工具调用 ID 维度拼接成完整字符串后再交给上层。
  事件流中保留 text / thinking 增量与工具调用事件的发生顺序。

- **F12：Round 1 触发条件**
  用户每次提交 prompt 触发 Round 1 LLM 请求，请求体携带全部已注册
  工具元信息。Round 1 流式响应可能包含：纯文本（无工具调用）、纯
  工具调用、文本与工具调用混合三种形态。

- **F13：工具串行执行与确认**
  Round 1 流结束后，按模型给出的顺序串行执行所有 tool_use 块。每个
  写类工具（write/edit）和命令类工具（run）执行前必须向用户征求
  确认（终端打印参数概要 + 等待 y/n 输入）。读类工具（read/glob/
  search）自动执行无需确认。每个工具执行结果不论成功失败都被收集成
  tool_result 块。

- **F14：tool_results 回灌历史并发起 Round 2**
  所有 tool_use 执行完毕后，工具结果按对应 tool_use_id 包成
  tool_result 块，作为一条 user 角色消息追加到历史；随后发起
  Round 2 LLM 请求。Round 2 流式渲染最终答复。

- **F15：Round 2 中再次出现 tool_use 的硬停处理**
  如果 Round 2 流中包含 tool_use 块，本阶段不执行：text 部分正常
  渲染并以 assistant 消息正常入历史，tool_use 块从入历史的消息中
  剥离丢弃；流结束后终端追加一行灰字提示"模型在最终答复中还想调用
  工具 X / 共 N 个，本阶段不再继续；下一轮可以追问"。

### 历史结构与协议适配

- **F16：消息历史扩展**
  消息历史从原先的纯文本 content 升级为内容块列表（text / tool_use /
  tool_result）。assistant 消息可包含混合块；user 消息既可以是纯文本
  （用户输入）也可以是 tool_result 列表（工具回填）。Provider 在请求
  时按各自协议格式将历史序列化。

- **F17：协议特性差异处理**
  Anthropic 协议：tool_use 与 tool_result 走 content 块原生格式；
  OpenAI 协议：assistant 的工具调用走 tool_calls 字段，user 的工具
  结果走 role=tool 的独立消息。两种格式在 Provider 层完成翻译，对
  上层完全透明。

- **F18：thinking 与工具的兼容**
  Anthropic 协议下 thinking 开启时仍正常发起工具调用流程；thinking 块
  在历史中保留并在 Round 2 中如实回传给后端（如协议规范有要求）。
  OpenAI 协议下不带 thinking。

### UI 呈现

- **F19：工具执行可视化（中等粒度）**
  每次工具执行时在终端展示：
  - 调用前一行灰字："▸ <tool_name>(<关键参数概要>)"，参数过长时
    截断到约 80 字符
  - 写类/命令类工具：参数概要后停顿，提示用户 [y/N] 确认
  - 调用后一行灰字摘要：read 显示读取行数、glob/search 显示匹配数、
    run 显示退出码、write/edit 显示成功/失败，错误时显示错误类别
  完整工具结果不在终端打印（只发给模型），避免刷屏。

- **F20：用户拒绝的反馈**
  用户对确认提示输入 n / N / 回车（默认拒绝）后：终端打印一行"已拒绝
  执行 <tool_name>"，同时把"用户拒绝执行此工具"作为该 tool_use 的
  tool_result 反馈给模型，流程继续走 Round 2。

### 配置与依赖

- **F21：工具系统始终启用**
  本阶段不提供"关闭工具系统"的配置或命令；启动后所有 6 个工具自动
  注册，所有请求都携带 tools 列表。

## 非功能需求

### 性能与响应

- **N1：流式不被工具调用打断感**
  Round 1 流式过程中，文本/思考增量按既有方式逐字渲染；遇到工具调用
  事件时不立即打断模型流式，而是缓存到流结束后统一展示与执行。Round
  2 同样保持流式逐字渲染。

- **N2：工具执行不阻塞输入**
  工具串行执行期间，UI 应及时反馈每一步的状态（"调用中"、"完成"），
  避免界面长时间无响应感。Ctrl+C 应能在合理时间（秒级）中止当前工具
  执行：尚未完成的 tool_use 当作"用户中断"返回；剩余未执行的 tool_use
  全部跳过；不进入 Round 2，整个 turn 视作"被用户取消"。

### 可用性

- **N3：错误信息对模型友好**
  工具执行失败时返回的错误描述应让模型有机会自行调整：包含错误类别
  （如"路径越界"、"未找到匹配"、"匹配 N 次需更多上下文"、"超时"等）
  与具体细节，避免只给"出错了"之类无信息量的反馈。

- **N4：终端兼容性继承**
  本阶段引入的所有 UI 元素（工具调用提示、确认提示、简略反馈、灰字
  提示）必须沿用第一阶段的"朴素 sys.stdout.write + 简单 ANSI 颜色"
  策略，保证在 Windows PowerShell 5.x 经典 conhost 上不出现控制
  字符泄漏。

- **N5：Ctrl+C 与确认提示**
  在工具确认提示等待用户输入 y/N 期间按 Ctrl+C 等同于"拒绝" + "中断
  整个 turn"：当前工具不执行、剩余 tool_use 跳过、不进入 Round 2、
  回到主输入提示符。

### 可维护性

- **N6：模块边界清晰**
  Tool 抽象与具体工具实现集中在 `tools` 模块；注册中心独立维护；
  Provider 层承担"协议→工具元信息序列化"与"SSE → 工具调用事件
  反序列化"两个翻译职责；chat 层负责调度（确认 → 执行 → 回灌 →
  Round 2）；REPL 不感知工具细节。

- **N7：工具扩展成本**
  新增一个工具的所需改动应限定在：① 在 tools 目录下新增一个文件
  实现 Tool 子类；② 在 tools 注册段追加一行注册。Provider、chat、
  REPL、render 等其他模块的代码均无需修改。

- **N8：协议扩展成本继承**
  本阶段引入的工具调用支持需在两种协议（Anthropic / OpenAI）下都
  实现；后续新增 wire protocol 时，新 Provider 子类需自行翻译工具
  元信息与解析工具调用流，但 Tool 抽象与上层逻辑保持不变。

- **N9：中文注释**
  本阶段新增的所有源码注释、docstring、用户可见提示文本均使用中文，
  与第一阶段一致。

### 安全与约束

- **N10：工作目录沙盒**
  路径相关工具的所有路径都必须经过"解析为绝对路径并校验落在 CWD 子树
  内"的统一前置检查；越界绝不执行。run 工具的子进程以 CWD 为工作
  目录启动，不允许通过参数指定其他 cwd。

- **N11：危险操作必经确认**
  write / edit / run 三类工具的每一次调用执行前都必须经过用户确认。
  本阶段不提供"全部允许"开关或"会话内记住允许"的便利路径——确认
  提示就是最后一道闸门。

- **N12：敏感信息保护继承**
  api_key 不得出现在任何工具相关的输出中（工具调用提示、确认提示、
  错误信息、模型可见的 tool_result 等）。即便用户向模型问起 api_key，
  模型从历史中也只看到 ProviderConfig 的非敏感字段（已由第一阶段保证）。

### 兼容性与依赖

- **N13：Python 版本继承**
  Python 3.10 及以上，与第一阶段一致。

- **N14：依赖最小化**
  本阶段所引入的第三方依赖应限定在第一阶段已有的集合内
  （prompt_toolkit、rich、PyYAML、httpx）；不引入新的运行时依赖。
  search/glob 的具体实现技术与 diff 渲染方案由 plan.md 决定，但需
  优先选用 Python 标准库。

## 不做的事

本阶段明确**不**包含以下能力，留待后续迭代：

### Agent 循环

- 不做 Round 2 之后的多轮循环：Round 2 的回复无论是否含 tool_use 都
  视为本 turn 终点
- 不做"工具调用上限"机制（如最多 N 次循环）——因为根本不进入循环
- 不做子代理（sub-agent）/ 任务分解
- 不做工具调用之间的依赖推理（哪个先调、哪个后调由模型决定，本阶段
  按模型给出的顺序串行）

### 工具能力

- 不做更多工具：本阶段固定 6 个（read、write、edit、run、glob、search），
  不做 web 搜索、HTTP 请求、git 操作、todo list、日历等扩展工具
- 不做工具的并行执行
- 不做 MCP（Model Context Protocol）工具协议适配
- 不做用户自定义工具（运行时通过 yaml 或脚本注册自定义工具）

### 文件相关工具

- 不支持二进制文件（read 解码失败时返回错误，不做 base64 编码回传）
- 不支持图片/PDF 等多模态文件读取
- 不做文件 watcher（监视文件变化主动通知模型）
- write 工具不做"已存在则备份原文件"
- edit 工具不做模糊匹配/正则替换/多片段替换；只做单段原文唯一匹配
- glob 不支持自定义忽略规则（写死的噪声目录列表）
- search 不支持二进制文件搜索、不支持替换（只搜不改）

### 命令工具

- run 工具不做交互式命令支持（不接 stdin、不能边运行边输入）
- run 不做后台进程管理（启动后台任务、查看任务列表等）
- run 不做命令白名单/黑名单
- run 不做"流式输出"（命令输出不实时显示给模型，只在结束时一次性返回）

### 安全与权限

- 不做"会话内记住允许某工具"的便利路径——每次 write/edit/run 都要确认
- 不做 yolo 模式 / `--auto-approve` 启动开关
- 不做基于路径的细粒度权限（如"只允许写 src/ 下的文件"）
- 不做工具执行的资源限制（CPU、内存、磁盘配额）
- 不做命令沙箱（容器、chroot）

### UI 与体验

- 不做工具调用的进度条 / spinner（中等粒度 UI 已够用）
- 不做工具结果的折叠/展开交互
- 不做完整的 diff 高亮渲染（edit 确认提示用文本 diff，不彩色）
- 不做工具调用的复盘视图（"这次 turn 调用了哪些工具"汇总）
- 不做工具结果输出的 Markdown 格式化

### 历史与持久化

- 不做工具调用历史的持久化（继承第一阶段：进程退出后历史消失）
- 不做"重发上一个工具调用"命令
- 不做 token 预算管理（让模型知道还剩多少上下文）

### 协议层

- 不引入 MCP / OpenAI Assistants API / Anthropic Agent SDK 等高层封装
- 不做 prompt caching / 工具元信息缓存
- 不做工具元信息的多语言/i18n 描述

## 验收标准

每条 AC 通过运行代码或观察终端行为来验证；与具体实现解耦。

### Tool 抽象与注册

- **AC1：Tool 抽象具备最小契约**
  阅读代码确认 Tool 抽象定义了 name / description / parameters_schema /
  execute 四项；6 个具体工具（read/write/edit/run/glob/search）均
  继承该抽象并实现全部成员。

- **AC2：工具注册中心可按名查找**
  启动 mewcode 后，通过测试脚本调用注册中心的"按名查找"接口能取
  到全部 6 个工具；查找不存在的名字返回空（或抛 KeyError，由 plan
  决定）。

- **AC3：工具注册中心按协议格式输出元信息**
  注册中心提供"序列化为 Anthropic 格式"和"序列化为 OpenAI 格式"两
  个出口；输出符合两种协议的 tools 字段规范（Anthropic：
  `{name, description, input_schema}`；OpenAI：
  `{type:"function", function:{name, description, parameters}}`）。

- **AC4：扩展性可见**
  阅读代码确认新增工具只需 ① tools 目录加新文件 ② tools 注册段加
  一行；REPL、chat、Provider、render、commands 模块的代码均无需修改。

### 6 个核心工具

- **AC5：read 基础读取**
  在 tmp 目录创建 hello.py 写入若干行内容；通过测试脚本直接调用
  read 工具，返回的 text 中包含原文每一行。

- **AC6：read offset/limit 按行读取**
  读 100 行的文件指定 offset=10、limit=5，返回的 text 仅含第 11~15
  行（行号语义 1-based）。

- **AC7：read 大文件截断**
  生成一个 500KB 的文本文件；read 工具返回的 text 长度不超过 256KB
  且包含"已截断"提示。

- **AC8：write 新建文件**
  在 REPL 中让模型写一个新文件 demo.txt；终端弹出确认提示展示路径
  与首若干行内容，输入 y 后文件创建成功，再次 read 能取到新内容。

- **AC9：write 用户拒绝**
  让模型写文件，确认提示输入 n；文件未创建；模型在 Round 2 中收到
  "用户拒绝执行此工具"反馈，最终答复体现这一信息（如说明放弃此操作）。

- **AC10：edit 唯一匹配替换**
  在文件中放入唯一字符串 ALPHA；让模型把 ALPHA 改成 BETA；确认 y 后
  read 文件能看到 BETA、找不到 ALPHA。

- **AC11：edit 匹配多次报错**
  在文件中放入 3 个 ALPHA；让模型把 ALPHA 改成 BETA；工具返回结构化
  错误"匹配 3 次需更多上下文"，文件保持不变；模型在 Round 2 中据此
  调整或说明。

- **AC12：edit 未匹配报错**
  让模型把不存在的 GAMMA 改成 DELTA；工具返回结构化错误"未找到匹配"，
  文件保持不变。

- **AC13：run 命令成功**
  让模型跑 `python --version`（或 `echo hello`）；确认 y 后工具返回
  含 stdout 与 exit_code=0；终端显示 "退出码 0"。

- **AC14：run 命令失败**
  让模型跑 `python -c "import sys; sys.exit(7)"`；确认 y 后返回
  exit_code=7；模型在 Round 2 中体现这一信息。

- **AC15：run 超时**
  让模型跑一个会阻塞 90 秒的命令；60 秒内子进程被终止，工具返回结构
  化错误"超时"。

- **AC16：glob 基础匹配**
  在 CWD 准备若干 .py 与 .md 文件；让模型 glob `**/*.py`；返回的
  匹配列表只包含 .py 文件、按字母序排列。

- **AC17：glob 自动排除噪声目录**
  在 CWD 下准备 `__pycache__/x.py`、`.git/y.py`、`node_modules/z.py`；
  glob `**/*.py` 时这三类目录中的文件不出现在结果中。

- **AC18：search 基础匹配**
  在多个 .py 文件中放入唯一标记 NEEDLE_XYZ；search 该字符串能返回
  所有命中的（文件、行号、行内容）三元组。

- **AC19：search 单行截断**
  在文件中放入一个超过 500 字符的长行（含 NEEDLE_XYZ）；search 返回
  的该匹配行内容长度不超过 500 字符。

### 路径边界

- **AC20：路径越界拒绝执行**
  让模型 read `/etc/passwd`（或 `C:\Windows\System32\drivers\etc\hosts`）
  或 `../../../some_file`；工具返回结构化错误"路径越界"，未真正读取，
  模型在 Round 2 中据此说明无法访问。

- **AC21：CWD 内绝对路径可访问**
  当前 CWD 是 `e:\AI\vscode_project\mecode`；read 绝对路径
  `e:\AI\vscode_project\mecode\README.md` 工作正常（边界规则只看
  解析后的最终路径是否在 CWD 子树内）。

### 单轮闭环

- **AC22：Round 1 + 工具 + Round 2 完整闭环**
  发起 prompt "读一下 README.md 然后告诉我项目叫什么"；终端显示：
  Round 1 流式（可能很短的引言）→ ▸ read(README.md) 提示 → 工具简略
  反馈（"读取 N 行"）→ Round 2 流式最终答复（含 README 中的项目名）。

- **AC23：Round 1 无工具调用直接答复**
  发起 prompt "你好"；模型 Round 1 直接给文本回复；不进入 Round 2；
  行为与第一阶段一致（流式 + 用量行）。

- **AC24：单 turn 多个 tool_use 串行执行**
  发起 prompt "读 README.md 和 mewcode.yaml.example 这两个文件，告诉
  我各自的第一行"；Round 1 产出 2 个 read 调用；终端按顺序执行两个
  read 并展示两条简略反馈；Round 2 答复中包含两个文件的第一行。

- **AC25：Round 2 中再含 tool_use 硬停**
  人为构造场景使 Round 2 输出含 tool_use（如难以稳定复现，允许通过
  mock 响应或单元测试验证）；终端显示 Round 2 文本部分正常入历史，
  最后追加一行灰字提示"模型在最终答复中还想调用工具 X，本阶段不再
  继续"；下一轮 prompt 发送时不报"孤儿 tool_use"协议错误。

### 协议双支持

- **AC26：Anthropic 协议工具调用**
  默认供应商（deepseek-anthropic）下完成 AC22 完整闭环；
  thinking on / off 两种状态都验证一次。

- **AC27：OpenAI 协议工具调用**
  `/provider deepseek-openai` 切换后完成 AC22 完整闭环；终端可见
  Round 1 → tool 执行 → Round 2 流程。

### 用户体验

- **AC28：调用前简略提示**
  任意工具调用前终端打印一行 "▸ <tool_name>(<参数概要>)"；参数过长
  时截断到约 80 字符。

- **AC29：调用后简略反馈**
  read 显示"读取 N 行"；glob/search 显示"匹配 N 项"；run 显示"退出码 N"；
  write/edit 显示"成功"或错误类别；与 AC28 同样为单行灰字。

- **AC30：完整工具结果不刷屏**
  让模型 read 一个 200 行文件；终端只显示"读取 200 行"简略反馈，不
  原样打印 200 行内容；Round 2 答复中模型的总结里能体现文件内容。

- **AC31：write/edit/run 确认提示**
  这三类工具调用前终端显示参数概要并停顿，等待 y/N 输入；输入 y 执行、
  输入 n 或回车默认拒绝。

### 中断与错误

- **AC32：工具执行中 Ctrl+C**
  让模型跑 `python -c "import time; time.sleep(30)"`；确认 y 后命令开始
  执行；按 Ctrl+C；当前命令被终止；不进入 Round 2；回到 `>` 提示符；
  无 traceback 渗漏。

- **AC33：确认提示中 Ctrl+C**
  write/edit/run 确认提示等待 y/N 时按 Ctrl+C；当前 turn 整体取消、
  剩余 tool_use 跳过、不进入 Round 2、回到 `>` 提示符；无 traceback。

- **AC34：工具失败模型可调整**
  让模型 edit 一个文件中"匹配 3 次"的字符串；Round 2 模型应能据
  错误反馈说明"该字符串出现多次，请告诉我具体哪一处"或自行调整。

### 用量、安全

- **AC35：用量行覆盖 Round 1 + Round 2**
  完整闭环结束后，终端显示一行用量；该行 token 计数应反映两次 LLM
  请求的累计或最后一次（plan.md 决定）。

- **AC36：api_key 不回显**
  在工具相关的所有终端输出（调用提示、确认提示、简略反馈、错误信息）
  中，api_key 的实际值不出现。

### 兼容性

- **AC37：第一阶段功能不退化**
  纯对话场景（不触发 tool_use）的全部第一阶段验收行为（流式、多轮、
  /clear、/think on/off、/provider 切换、/exit、Ctrl+C 双击退出等）
  保持通过。

- **AC38：Windows 终端兼容**
  在 Windows PowerShell 5.x 下完成 AC22 闭环，无 `?[2K` 类乱码、
  无 traceback 渗漏；Markdown 字符按朴素文本显示（继承第一阶段降级）。
